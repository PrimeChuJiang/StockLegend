# 2D 卡牌界面实现方案

## 概述

桌面上的每张卡牌是一个 `Node2D` 场景节点，**直接持有**自己对应的数据对象引用（`Stock` 或 `Article`）。所有卡牌共享一个基类 `CardNode`，负责通用的视觉行为；子类负责各自的数据绑定和显示更新。

---

## 类结构

```
CardNode (Node2D)
│
│  共用职责：
│  ├── 翻转动画、拖拽、悬浮高亮、点击响应
│  ├── 卡面纹理/Sprite 管理
│  └── 鼠标交互区域（Area2D / CollisionShape）
│
├── StockCardNode extends CardNode
│   │
│   │  独有数据：
│   │  └── var stock: Stock       ← 直接引用运行时股票对象
│   │
│   │  独有职责：
│   │  ├── 显示股票名、当前价格、涨跌幅、情绪指示
│   │  ├── 监听 GameBus.stock_price_changed → 刷新价格显示
│   │  ├── 监听 GameBus.sentiment_modifier_applied → 刷新情绪指示
│   │  └── 监听 GameBus.stock_delisted → 视觉标记退市
│   │
│   │  数据访问方式：
│   │  └── 直接读 stock.current_price / stock.get_sentiment() / stock.def.name
│   │
│   └── setup(p_stock: Stock)     ← 生成时由场景注入
│
└── ArticleCardNode extends CardNode
	│
	│  独有数据：
	│  └── var article: Article   ← 直接引用运行时文章对象
	│
	│  独有职责：
	│  ├── 显示文章类型、影响力、可信度、目标行业/股票
	│  ├── 显示剩余新鲜度（草稿态）或剩余持续回合（已发表态）
	│  ├── 监听 GameBus.article_expired → 卡牌消失动画
	│  └── 监听 GameBus.article_busted → 被打假视觉效果
	│
	│  数据访问方式：
	│  └── 直接读 article.final_impact / article.freshness / article.is_published
	│
	└── setup(p_article: Article) ← 生成时由场景注入
```

## 生命周期对应关系

### Stock（安全）

`StockManager`（AutoLoad）持有所有 Stock 实例，整局游戏存活，永远不会先于卡牌节点被释放。卡牌节点直接引用 `stock` 是安全的。

### Article（需同步销毁）

Article 的生命周期分两个阶段：

- **草稿态**：`PlayerState.draft_articles` 持有，过期时被移除
- **已发表态**：需要有一个已发表文章列表持有

卡牌节点应在数据对象被移除时一同销毁。流程如下：

```
WorldEndActor 结算：article.freshness 归零
  → player_state.draft_articles.erase(article)
  → GameBus.article_expired.emit(article)
	  → ArticleCardNode 收到信号
		  → 对比 self.article == 信号传来的 article
		  → 匹配 → 播放消失动画 → queue_free()
```

数据对象（Article）虽然从列表中移除了，但因为 `ArticleCardNode` 还持有引用，`RefCounted` 引用计数不会归零，Article 不会被释放。等 `queue_free()` 执行后，卡牌节点销毁，引用释放，Article 才真正回收。这正好是我们需要的——卡牌在动画期间仍然能读取 article 的数据。

## 信号驱动更新

卡牌节点**不轮询**数据，而是通过 `GameBus` 信号被动刷新。每种卡牌只监听自己关心的信号：

### StockCardNode 监听

| 信号 | 过滤条件 | 响应行为 |
|------|---------|---------|
| `stock_price_changed(stock_id, old, new)` | `stock_id == stock.def.id` | 刷新价格标签、涨跌箭头颜色 |
| `sentiment_modifier_applied(stock_id, mod)` | `stock_id == stock.def.id` | 刷新情绪图标/光环 |
| `stock_delisted(stock_id)` | `stock_id == stock.def.id` | 灰度化卡面、禁用交互 |

### ArticleCardNode 监听

| 信号 | 过滤条件 | 响应行为 |
|------|---------|---------|
| `article_expired(article)` | `article == self.article` | 消失动画 → `queue_free()` |
| `article_busted(article)` | `article == self.article` | 碎裂/打叉动画 |

信号回调中用 `stock_id` 或 `article` 引用做**过滤**，确保每张卡只响应自己的数据变化。

## 场景生成流程

某个管理桌面布局的节点（比如 `TableManager` 或场景根脚本）负责生成卡牌。

### 游戏开局，铺设股票卡

```gdscript
for stock in StockManager.get_all_stocks():
	var card := StockCardScene.instantiate()
	card.setup(stock)                          # 注入引用
	card.position = _next_stock_slot_position() # 决定桌面位置
	table.add_child(card)                       # 挂到场景树
```

### 玩家发表文章时，生成文章卡

```gdscript
GameBus.article_published.connect(func(article, channel):
	var card := ArticleCardScene.instantiate()
	card.setup(article)                         # 注入引用
	card.position = _next_article_slot_position()
	table.add_child(card))
```

注入引用的时机就是 `setup()` 调用的时刻。之后卡牌节点自己管自己——通过持有的引用读数据，通过信号知道何时刷新。场景管理者不需要维护"卡牌节点 - 数据"的映射表。

## 扩展新卡牌类型

新增卡牌类型（比如环境事件卡）只需新建子类，不影响已有代码：

```
CardNode
├── StockCardNode
├── ArticleCardNode
└── EventCardNode extends CardNode      ← 新增
	├── var event: ScheduleEventConfig
	├── setup(p_event: ScheduleEventConfig)
	└── 监听 GameBus.event_revealed → 翻牌动画
```

## 设计决策总结

| 考量 | 决策 | 理由 |
|------|------|------|
| 数据访问 | 直接引用 | 不经过字典查询，读属性就是一次成员访问 |
| 更新时机 | 信号驱动 | 不轮询，不需要每帧检查数据是否变化 |
| 耦合度 | 仅依赖数据对象 + GameBus | 卡牌节点不依赖任何其他场景节点 |
| 生命周期 | Stock 由 AutoLoad 保证存活；Article 由 RefCounted 引用计数保证卡牌动画期间不被释放 | 安全且符合 Godot 内存管理模型 |
| 扩展性 | 继承 CardNode 新增子类 | 不影响已有代码 |
